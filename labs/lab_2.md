## Контрольные вопросы для самопроверки
- Какой формат имени сетевого ресурса может использоваться при
обращении к нему?

Форматы имени сетевого ресурса, используемые при обращении к нему:

UNC (Universal Naming Convention): \\сервер\ш_share\путь\файл — для SMB/CIFS в Windows.
URI/URL: схема://хост[:порт]/путь[?query][#фрагмент] — для HTTP, FTP, smb:// и др.
FQDN (полное доменное имя): host.example.com — используется в DNS.
IP-адрес: IPv4 (192.0.2.1) или IPv6 ([2001:db8::1]) — можно указывать вместо имени хоста.
NetBIOS имя: NETBIOSNAME — старый формат для локальных сетей Windows.
NFS-путь: сервер:/экспорт/путь — для NFS.
Samba/SMB URL: smb://сервер/шare/путь — альтернативный URL-формат для SMB.
LDAP DN (Distinguished Name): cn=User,dc=example,dc=com — для доступа к LDAP-ресурсам.
S3-стиль: s3://bucket/key или https://bucket.s3.amazonaws.com/key — для объектов в S3-совместимых хранилищах.


- Какой протокол необходим для работы с утилитой ping? Найти описание
и характеристики протокола.

## Описание

- Назначение: обмен служебными сообщениями между узлами IP-сети для диагностики и контроля (информирование о ошибках, проверка доступности хоста).
- Стандарт: определён в RFC 792 (ICMPv4). Для IPv6 — ICMPv6 (RFC 4443 и др.).
## Типы и сообщения
- Сообщения контроля/ошибок: Destination Unreachable, Time Exceeded, Redirect, Source Quench (устаревшее) и др.
- Сообщения запроса/ответа: Echo Request и Echo Reply — именно они используются утилитой ping.
## Формат пакета (ICMPv4, упрощённо)
- Поле Type (1 байт) — тип сообщения (например, 8 = Echo Request, 0 = Echo Reply).
- Поле Code (1 байт) — уточняющий код типа.
- Поле Checksum (2 байта) — контрольная сумма ICMP-сообщения.
- Данные (variable) — для Echo: 2-байтовый идентификатор, 2-байтовый номер последовательности и полезная нагрузка.
## Взаимодействие с IP
- ICMP — протокол уровня сети (работает поверх IP). - - ICMP-сообщение помещается в поле данных IP-пакета (протокол number 1 для IPv4, 58 для IPv6).
- Для отправки/приёма ICMP часто требуются привилегии (raw sockets).
## Характеристики и поведение
- Без установки соединения: сообщения одиночные, без установки сессии.
- Не ориентация на доставку данных приложениям — служит для диагностики и обмена ошибками.
- Echo Request/Reply используются для измерения доступности и времени отклика (RTT).
- Может быть ограничен или фильтроваться брандмауэрами (часто блокируют ICMP или конкретные типы).
- Уязвимости/ограничения: возможны атаки типа ICMP flood (DDoS), отражённые атаки; поэтому в сети ICMP может быть частично/полностью отключён.
## Типичные параметры измерений (что возвращает ping)
- Время отклика (RTT) — обычно в миллисекундах.
- Потери пакетов — процент недоставленных эхо-ответов.
- TTL (Time To Live) — значение поля IP, которое помогает оценить количество промежуточных узлов.

- Зачем используется параметр all в утилите ipconfig?

Параметр /all в утилите ipconfig (Windows) выводит полную подробную информацию о сетевой конфигурации всех интерфейсов вместо краткого обзора. В частности показывает:

- Полный список всех сетевых адаптеров и их состояний.
- Физический (MAC) адрес каждого адаптера.
- Полные IPv4 и IPv6 адреса, маски и шлюзы.
- Значения DHCP: включён/выключен, адрес сервера DHCP, время аренды (Lease Obtained/Lease Expires).
- DNS-серверы и порядок их использования.
- WINS-серверы и NetBIOS-статус.
- Подробности туннельных адаптеров и виртуальных интерфейсов.
- Другая диагностическая информация (например, DHCP-имя клиента).
Использование: ipconfig /all — полезно при диагностике сети или при необходимости скопировать все параметры конфигурации.

- Каким образом утилиты ping и tracert осуществляют прослеживание
маршрутов пакетов к заданному узлу?

Обе утилиты используют поле TTL (Time To Live) в заголовке IP и ответы ICMP для пошагового определения промежуточных узлов.

## Кратко по шагам:

- Поле TTL: при прохождении каждого маршрутизатора TTL уменьшается на 1; при достижении 0 пакет отбрасывается, и маршрутизатор отправляет обратно ICMP Time Exceeded.

- Tracert/traceroute:
-      Посылает пакеты с возрастающим TTL: сначала TTL=1, затем TTL=2 и т.д.
-       Каждый отправленный пакет с TTL=n достигает n-го хопа, где при обнулении TTL маршрутизатор возвращает ICMP Time Exceeded; утилита фиксирует IP (и обратное DNS) и время.
-       Продолжают увеличивать TTL пока не получат ответ от целевого узла (ICMP Echo Reply в Windows tracert или ICMP Port Unreachable при UDP-traceroute), либо пока не достигнут заданного максимума.
-       Обычно по каждому TTL отправляют несколько попыток (например, 3) для измерения вариативности RTT.
- Ping:
-       Непосредственно не предназначен для трассировки, но использует ICMP Echo Request/Reply для проверки доступности и измерения RTT.
-       Некоторые реализации (например, Windows tracert отдельная утилита) используют тот же механизм ICMP; команда ping с опцией для установки TTL может быть использована вручную для имитации одного шага traceroute (посылать Echo Request с определённым TTL и смотреть, откуда придёт Time Exceeded).

## Особенности и ограничения:

- Разные реализации: Windows tracert по умолчанию использует ICMP Echo Request; UNIX traceroute часто использует UDP по умолчанию (целевой хост возвращает ICMP Port Unreachable), есть опции для ICMP/TCP.
- Маршрутизаторы или брандмауэры могут блокировать ICMP Time Exceeded или ответы → появятся звёздочки (*) или пропуски.
- Ассиметричные маршруты и NAT могут искажать отражённую информацию; ответный путь может отличаться от прямого.

- Можно ли утилитой tracert задать максимальное число ретрансляций?

Да. В Windows tracert можно задать количество попыток (ретрансляций) для каждого значения TTL с помощью ключа -w (таймаут в миллисекундах) — но прямого параметра «число попыток» нет; по умолчанию tracert посылает 3 запроса на каждый хоп и это поведение не настраивается стандартными опциями.

- Что такое localhost?

localhost — специальное имя хоста, обозначающее текущий компьютер (loopback-интерфейс). Обычно разрешается в IP 127.0.0.1 для IPv4 и ::1 для IPv6; используется для тестирования и связи процессов на одной машине без выхода в сеть.

для тестирования связи), будет происходить постоянная отправка пакетов и
можно обнаружить ситуацию, при которой появляется или пропадает связь.

или установление соединения). Использование адреса 127.0.0.1 позволяет

Определить соответствие локального IP-адреса, физическому
(аппаратному) адресу в локальной сети.
Возможность просматривать и изменять ARP-таблицу, в которой
хранятся пары «МАС-адрес - IP-адрес» для тех узлов, с которыми в недавнем
происходил обмен данными, дает утилита ARP. Эта таблица формируется
автоматически при работе сетевого узла, но администратор сети может
вносить в нее записи вручную.
Узел, собирающийся отправить сообщение другому узлу, должен
предварительно узнать MAC адрес получателя сообщения. Для решения
данной задачи узел применяет технологию ARP, отправляя запрос узлам своей
локальной сети. Данный ARP запрос содержит IP адрес получателя. Из всех
узлов, получивших данный запрос, отвечает лишь тот, у кого
соответствующий IP адрес. В своем ответе (ARP отклике) этот узел сообщает
свой MAC адрес. И лишь после этого первый узел сможет отправить ему свое
сообщение. Компьютеры чаще всего отправляют свои сообщения
маршрутизатору и, следовательно, в своих ARP запросах они указывают
адрес основного шлюза. Для уменьшения ARP трафика компьютеры хранят в
своей памяти таблицу с IP и MAC адресами тех устройств, с которыми они в
последнее время обменивались сообщениями. Управление работой утилиты
возможно с помощью ключей (перечень выводится на экран командой arp).
Утилита ARP с ключом -a позволяет вывести на экран всю ARP-таблицу.
